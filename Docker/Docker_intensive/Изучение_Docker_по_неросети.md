## 1. Контейнер vs Виртуальная Машина

### 1.1 Основное отличие

| Характеристика | Виртуальная Машина (VM) | Контейнер (Docker) |
| :--- | :--- | :--- |
| **Что эмулирует** | Железо (CPU, RAM, диск) | Пользовательское пространство ОС |
| **Ядро** | Своё собственное (Guest OS) | **Общее с хостом** (ядро Linux) |
| **Размер** | Гигабайты (ГБ) | Мегабайты (МБ) |
| **Время запуска** | Минуты | Миллисекунды |
| **Изоляция** | Полная (уровень гипервизора) | Процессная (уровень ядра) |
| **Производительность** | Накладные расходы 5-15% | Накладные расходы ~1-2% |

### 1.2 Визуальная схема

```
┌─────────────────────────────────────────────────────────────┐
│                    ВИРТУАЛЬНЫЕ МАШИНЫ                        │
├─────────────────┬─────────────────┬─────────────────────────┤
│   App A         │   App B         │   App C                 │
│   Bin/Libs      │   Bin/Libs      │   Bin/Libs              │
│   Guest OS      │   Guest OS      │   Guest OS              │
├─────────────────┴─────────────────┴─────────────────────────┤
│                    Гипервизор (Hypervisor)                   │
├─────────────────────────────────────────────────────────────┤
│                    Host OS (Ядро)                            │
├─────────────────────────────────────────────────────────────┤
│                    Инфраструктура                            │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    КОНТЕЙНЕРЫ (Docker)                       │
├─────────────┬─────────────┬─────────────┬───────────────────┤
│   App A     │   App B     │   App C     │   App D           │
│   Bin/Libs  │   Bin/Libs  │   Bin/Libs  │   Bin/Libs        │
├─────────────┴─────────────┴─────────────┴───────────────────┤
│                    Docker Engine                             │
├─────────────────────────────────────────────────────────────┤
│                    Host OS (Ядро)                            │
├─────────────────────────────────────────────────────────────┤
│                    Инфраструктура                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 Ключевой вывод

> **Контейнер — это просто процесс Linux**, который "думает", что он один в системе. У него нет своего ядра, он использует ядро хоста.

## 2. Три кита контейнеризации в Linux

Docker — это "обертка" над тремя технологиями ядра Linux.

### 2.1 Namespaces (Изоляция)

**Namespaces** делают так, чтобы процесс внутри контейнера не видел процессы снаружи.

| Тип Namespace | Что изолирует | Пример |
| :--- | :--- | :--- |
| `PID` | Процессы | Внутри контейнера твой процесс имеет PID 1 |
| `NET` | Сеть | Свой IP, интерфейсы, таблицы маршрутизации |
| `MNT` | Файловая система | Свои точки монтирования (`/`) |
| `UTS` | Hostname | Свое имя хоста |
| `IPC` | Межпроцессное взаимодействие | Свой доступ к shared memory |
| `USER` | Пользователи | Можно быть `root` внутри, но обычным пользователем снаружи |

**Проверка на практике:**
```bash
# Внутри контейнера
docker run --rm alpine ps aux
# Видим только процессы контейнера (обычно 1-3 процесса)

# На хосте
ps aux | grep sleep
# Видим реальный процесс с PID хоста (например, 12345)
```

### 2.2 Cgroups (Ограничения)

**Control Groups** не дают контейнеру сожрать все ресурсы хоста.

**Что можно ограничить:**
- Память (`--memory`)
- CPU (`--cpus`, `--cpu-shares`)
- Дисковый I/O (`--blkio-weight`)
- Network bandwidth (через tc)

**Пример:**
```bash
docker run -d --name limited \
  --memory="512m" \
  --cpus="1.0" \
  nginx
```

**Важно:** Если контейнер превысит лимит памяти → ядро убивает процесс (**OOM Kill**).
- **Код выхода:** `137` (128 + 9 SIGKILL)
- **Проверка:** `docker inspect <container> --format='{{.State.OOMKilled}}'`

### 2.3 UnionFS (Слои)

**Union File Systems** (обычно `overlay2`) позволяют образам состоять из слоев.

**Принцип работы:**
1. **Образ** = набор слоев только для чтения (базовый образ + твои изменения)
2. **Контейнер** = образ + **тонкий записываемый слой** сверху
3. **Copy-on-Write:** При изменении файла он копируется из нижнего слоя в верхний, и только потом меняется

**Схема:**
```
┌─────────────────────────────────────┐
│  Container Layer (Read/Write)       │  ← Все изменения здесь
├─────────────────────────────────────┤
│  Image Layer 3 (Read-Only)          │  ← Твой Dockerfile (CMD)
├─────────────────────────────────────┤
│  Image Layer 2 (Read-Only)          │  ← Твой Dockerfile (COPY)
├─────────────────────────────────────┤
│  Image Layer 1 (Read-Only)          │  ← Базовый образ (ubuntu)
└─────────────────────────────────────┘
```

**Важно:** При удалении контейнера записываемый слой **удаляется безвозвратно**. Данные в нем не сохраняются!

## 3. Архитектура Docker

### 3.1 Компоненты

```
┌─────────────────┐
│  docker (CLI)   │  ← Ты здесь. Отправляет команды через API.
└────────┬────────┘
         │ REST API / Unix Socket (/var/run/docker.sock)
         ▼
┌─────────────────┐
│  dockerd        │  ← Мозг. Управляет образами, сетями, томами.
│  (Daemon)       │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  containerd     │  ← Низкоуровневый рантайм. Скачивает образы,
│                 │    запускает контейнеры.
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  runc           │  ← Создает namespace/cgroups и запускает процесс.
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   Ядро Linux    │  ← Реальная работа: изоляция, планировщик, драйверы.
└─────────────────┘
```

### 3.2 Описание компонентов

| Компонент | Назначение |
| :--- | :--- |
| `docker` (CLI) | Командная строка. Отправляет команды демону. |
| `dockerd` (Daemon) | Главный процесс. Управляет всеми объектами Docker. |
| `containerd` | Промежуточный слой. Управляет жизненным циклом контейнеров. |
| `runc` | Low-level рантайм. Создает namespaces и cgroups, запускает процесс. |
| `Ядро Linux` | Реализует изоляцию (namespaces) и ограничения (cgroups). |

### 3.3 Проверка архитектуры

```bash
# Посмотреть версию демона
docker version

# Посмотреть информацию о системе
docker info

# Найти корневую директорию Docker
docker info | grep "Docker Root Dir"
# Обычно: /var/lib/docker
```

## 4. Файловая система и слои

### 4.1 Где хранятся данные

| Объект | Расположение на хосте | Сохраняется при удалении |
| :--- | :--- | :--- |
| **Образы** | `/var/lib/docker/overlay2/` | ✅ Да |
| **Контейнеры** | `/var/lib/docker/overlay2/` + `/var/lib/docker/containers/` | ❌ Нет (записываемый слой) |
| **Volumes** | `/var/lib/docker/volumes/` | ✅ Да |
| **Bind Mounts** | Любая папка на хосте | ✅ Да (зависит от хоста) |

### 4.2 Жизненный цикл данных

```
1. docker run nginx
   → Создается записываемый слой контейнера

2. docker exec -it <container> bash
   → echo "hello" > /tmp/test.txt
   → Файл создан в записываемом слое

3. docker rm -f <container>
   → Записываемый слой УДАЛЕН
   → Файл test.txt УТЕРЯН безвозвратно
```

### 4.3 Решение: Volumes

Для сохранения данных используй **Volumes**:

```bash
# Создание тома
docker volume create my-data

# Запуск с томом
docker run -d -v my-data:/data nginx

# Данные сохраняются даже после удаления контейнера!
```

## 5. Базовые команды

### 5.1 Жизненный цикл контейнера

```bash
# Запуск
docker run -d --name <name> <image>         # -d = фон (detached)
docker run -it --name <name> <image> bash   # -it = интерактивно

# Просмотр
docker ps                                   # Активные контейнеры
docker ps -a                                # Все контейнеры
docker inspect <container>                  # Детальная информация (JSON)

# Управление
docker stop <name>                          # Корректная остановка (SIGTERM)
docker kill <name>                          # Мгновенное убийство (SIGKILL)
docker start <name>                         # Запуск остановленного
docker restart <name>                       # Перезапуск
docker rm <name>                            # Удаление (только остановленного)
docker rm -f <name>                         # Удаление с принудительной остановкой

# Вход и логи
docker exec -it <name> bash                 # Интерактивный вход
docker exec -it <name> sh                   # Если bash нет (alpine)
docker logs <name>                          # Вывод логов
docker logs -f <name>                       # Follow (как tail -f)
docker logs --tail 100 <name>               # Последние 100 строк

# Мониторинг
docker stats                                # CPU, RAM, NET, IO в реальном времени
docker top <name>                           # Процессы внутри контейнера
```

### 5.2 Работа с образами

```bash
docker images                               # Список образов
docker pull <image>                         # Скачать образ
docker rmi <image>                          # Удалить образ
docker history <image>                      # Показать слои образа
docker tag <old> <new>                      # Переименовать тег
docker save <image> -o file.tar             # Экспорт в tar
docker load -i file.tar                     # Импорт из tar
```

### 5.3 Очистка

```bash
# Осторожно! Эти команды удаляют данные
docker system prune                         # Остановленные контейнеры, неиспользуемые сети, кэш
docker system prune -a                      # + все образы, не используемые контейнерами
docker system prune --volumes               # + неиспользуемые тома (ОПАСНО!)

# Удаление конкретных объектов
docker container prune                      # Только контейнеры
docker image prune                          # Только образы
docker volume prune                         # Только тома
docker network prune                        # Только сети
```

### 5.4 Ресурсы и безопасность

```bash
# Ограничение ресурсов
docker run --memory="512m" --cpus="1.0" <image>

# Запуск не от root
docker run --user 1000:1000 <image>

# Read-only файловая система
docker run --read-only <image>

# Убрать привилегии
docker run --cap-drop ALL <image>
```

## 6. Best Practices

### 6.1 Общие рекомендации

| ✅ Делай | ❌ Не делай |
| :--- | :--- |
| Используй конкретные теги (`nginx:1.25`) | Используй `latest` в продакшене |
| Запускай от non-root пользователя | Запускай от root по умолчанию |
| Используй `.dockerignore` | Копируй весь контекст включая `.git` |
| Ограничивай ресурсы (`--memory`, `--cpus`) | Давай контейнерам неограниченные ресурсы |
| Используй volumes для данных | Храни данные в контейнере |
| Сканируй образы на уязвимости | Игнорируй безопасность |

### 6.2 Чек-лист перед деплоем

- [ ] Образ собран с конкретным тегом (не `latest`)
- [ ] Контейнер запускается от non-root пользователя
- [ ] Лимиты памяти и CPU установлены
- [ ] Данные хранятся в volumes, а не внутри контейнера
- [ ] Секреты не зашиты в образ (используй secrets/env)
- [ ] Образ просканирован на уязвимости (Trivy, Clair)
- [ ] `.dockerignore` настроен корректно

### 6.3 Типичные ошибки и решения

| Ошибка | Причина | Решение |
| :--- | :--- | :--- |
| **Exit Code 137** | OOM Kill (нехватка памяти) | Увеличь `--memory` или оптимизируй приложение |
| **Permission Denied** | Неправильные права на volume | Используй `--user` или `chown` на хосте |
| **Container сразу падает** | Процесс завершается (не демон) | Добавь `sleep infinity` или исправь CMD |
| **Нет сети** | Контейнеры в разных сетях | Используй `--network` или создай общую сеть |
| **Данные потеряны** | Удаление контейнера с данными | Используй volumes для важных данных |

## 📝 Глоссарий

| Термин | Определение |
| :--- | :--- |
| **Image** | read-only шаблон для создания контейнеров |
| **Container** | запущенный экземпляр образа |
| **Layer** | слой файловой системы образа |
| **Volume** | механизм хранения данных вне контейнера |
| **Namespace** | механизм изоляции ресурсов ядра Linux |
| **Cgroup** | механизм ограничения ресурсов ядра Linux |
| **Overlay2** | драйвер файловой системы по умолчанию в Docker |
| **Daemon** | фоновый процесс Docker (`dockerd`) |
| **Registry** | хранилище образов (Docker Hub, Harbor, etc.) |
